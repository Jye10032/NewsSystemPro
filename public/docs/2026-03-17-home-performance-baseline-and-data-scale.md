  1. 图表库代码层动态导入
  2. 更轻量的占位方式
  3. 避免让大面积骨架进入首屏关键区域

## 十一、移除 antd 强制共享包后的构建结果

测试时间：2026-03-18  
优化内容：

- 从 `vite.config.js` 中移除 `manualChunks.antd = ['antd']`
- 不再把所有 Ant Design 代码强行打进一个共享 chunk
- 让 Rollup 按实际页面依赖重新拆分首页与后台表格能力

### 1. 构建产物变化

优化前，首页相关构建中存在一个较大的共享 antd 包：

| Chunk | 体积 | gzip |
|-------|------|------|
| `antd-*.js` | `~921.92 kB` | `~291.81 kB` |

优化后，构建结果变为更细粒度拆分：

| Chunk | 体积 | gzip | 说明 |
|-------|------|------|------|
| `Home-096f237c.js` | `44.58 kB` | `15.26 kB` | 首页自身逻辑 |
| `Table-e343d2fc.js` | `166.01 kB` | `52.75 kB` | 表格能力独立拆出 |
| `Pagination-25e27183.js` | `23.78 kB` | `7.12 kB` | 分页能力独立拆出 |
| `index-7e224813.js` | `290.76 kB` | `95.84 kB` | 其余通用 antd 能力 |

### 2. 结论

- 这次调整的核心收益不是让 antd 消失，而是避免首页首访时被后台重型组件强绑定
- `Table` 已经被拆成独立 chunk，说明首页不再必须预先加载表格能力
- 这比“继续手动压 Home 组件代码”更有效，因为真实大头原本在共享依赖策略而不是业务 JSX

阶段判断：

- **应保留** 移除 `antd` 强制共享包的改动
- 后续如果继续优化首屏，应优先关注：
  1. 图表库 chunk 是否继续下沉
  2. 首页首屏是否仍被低频组件拖慢
  3. 登录后进入首页的真实冷启动指标

## 十二、静默请求改造记录

测试时间：2026-03-18  
优化内容：

- 为请求层增加 `skipGlobalLoading` 标记
- 首页聚合接口 `GET /api/dashboard/home` 改为静默请求
- 启动阶段的 `GET /api/users/me`、`GET /rights` 改为静默请求
- 请求日志只保留在开发环境，生产构建不再执行 `console.log`

### 1. 目的

- 避免首页缓存命中后的后台刷新继续驱动全局 loading 计数器
- 降低进入后台壳层时的无意义重渲染与右上角 loading 抖动
- 减少生产环境请求日志对主线程和控制台的噪音

### 2. 结论

- 这次改动属于运行时层面的“减噪”优化
- 它不会像分包那样直接反映为单个 chunk 体积下降
- 但它能让首页聚合缓存、用户信息回填、权限树回填这些非阻塞请求不再打断首屏稳定性

阶段判断：

- **应保留** 静默请求机制
- 后续可以继续把“后台预热型请求”和“用户显式等待型请求”拆开管理，而不是统一走一个全局 loading

## 十一、首页顶部首卡为什么会成为 LCP 元素

分析对象：`src/sandbox/home/Home.tsx`

### 1. 当前首卡结构

首页顶部第一张卡片包含：

- 用户头像与用户名
- 区域与角色
- “我的新闻 / 我的浏览 / 我的点赞”三项统计
- “查看个人数据分析”入口

这块区域具备三个典型特征：

1. 它处在首屏最上方
2. 它的可见面积大于普通文本行
3. 它的统计数字依赖异步接口返回后再更新

因此 Lighthouse 很容易把它识别成首屏最大的内容元素。

### 2. 它为什么会拖慢 LCP

虽然首页已经把图表做成了异步组件，但顶部首卡仍然会受到这类问题影响：

- 进入页面先渲染一版默认值：`0`
- `/api/dashboard/home` 返回后，再把三项个人统计更新成真实值
- 对 Lighthouse 来说，LCP 关注的是“最终稳定显示”的时间，而不是“占位先出现”的时间

所以即使页面已经有基础骨架，若首卡中的主要文本块或统计块在接口返回后发生更新，`LCP element render delay` 仍然会被拉高。

### 3. 这说明当前瓶颈已经变化

现在首页的主要问题不再是：

- 打包没拆开
- 图表库阻塞主线程

而更接近：

- 首屏关键文本区依赖异步数据后置更新
- 首屏最大元素恰好位于顶部首卡
- Lighthouse 把“首卡最终稳定时间”算进了 LCP

## 十二、针对顶部首卡的具体优化点

这部分优先级应高于继续微调图表。

### 1. 优先做首卡数据稳定化

目标不是“把首卡隐藏掉”，而是让它更早稳定。

可行方向：

- 将顶部首卡拆成独立组件，减少首页其它状态更新对它的连带重渲染
- 对三项个人统计使用固定宽度占位，避免数字回填时影响文本块稳定性
- 若接口未返回，可先展示轻量占位文本，例如 `--`

注意：

- 这里更适合“小面积稳定占位”
- 不适合再引入大面积骨架屏，因为前面的实验已经证明会拉低分数

### 2. 优先缓存 dashboard 聚合结果

如果用户是从登录页进入首页，可以考虑：

- 登录成功后预取 `GET /api/dashboard/home`
- 或在首页对该接口做短时内存缓存 / session 缓存

这样可以减少二次进入首页时首卡再次从 `0` 更新到真实值的过程。

### 3. 把“用户静态信息”和“统计数据”拆开看

当前首卡里：

- 用户名、区域、角色基本是同步可得的
- 个人统计完全依赖接口

更合理的思路是：

- 用户身份区立即稳定渲染
- 统计区单独更新，但保持固定布局和固定宽度

这样即使统计值稍后回填，也不容易让整个首卡继续充当 LCP 的最终完成点。

### 4. 继续观察 LCP element，而不是只看总分

后续每次优化首页时，除了看 `Performance` 总分，还应重点确认：

- `LCP element` 是否仍然是顶部首卡
- `element render delay` 是否下降
- 是否出现新的首屏大块元素替代它成为 LCP

## 十三、下一步建议

如果继续优化首页，建议顺序如下：

1. 先处理顶部首卡的稳定渲染问题
2. 再评估 dashboard 数据缓存
3. 最后再决定是否继续细调图表加载策略

原因：

- 当前 `TBT` 已经是 `0ms`
- 继续压 JS 对首页分数的边际收益在下降
- 顶部首卡才是当前更真实的 LCP 优化目标

因此，后续首页性能优化的重点，应从“如何继续压缩首包”，转向“如何减少首页不必要的数据请求、前端聚合和首屏关键区域的延后渲染”。

## 十四、首卡稳定化 + dashboard 缓存实验记录

测试时间：2026-03-17  
优化内容：

- 将首页顶部首卡拆成独立组件
- 三项个人统计在加载阶段使用固定宽度占位 `--`
- 为 `GET /api/dashboard/home` 增加 `memory + sessionStorage` 的短时缓存

### 1. 实验结果

本轮截图中出现了两次典型结果：

| 指标 | 结果 A | 结果 B |
|------|--------|--------|
| Performance | 94 | 93 |
| FCP | 0.8s | 0.8s |
| LCP | 0.9s | 1.4s |
| TBT | 0ms | 0ms |
| CLS | 0.026 | 0.027 |
| Speed Index | 2.2s | 1.6s |

### 2. 结论

- 这次优化**没有明显改善 FCP**
- 说明“页面第一次出现内容”的时间并不主要受首卡数据回填影响
- 这次优化更多是在减少：
  - 首卡内部异步回填造成的波动
  - 二次进入首页时重复从 `0` 回填真实值的问题

换句话说：

- 它不是一次“首屏开始更快”的优化
- 更像是一次“缩小 LCP 噪声源、继续定位瓶颈”的优化

### 3. 当前判断

结合这次实验和 Lighthouse 详情，可以得到更明确的判断：

- 当前首页不是白屏慢问题，因为 `FCP` 仍稳定在 `0.8s`
- 当前首页也不是主线程阻塞问题，因为 `TBT` 仍稳定在 `0ms`
- 当前更像是：
  - 首屏关键 chunk 下载链路较长
  - 首屏路由壳层与 UI 组件依赖较重
  - 导致最大内容元素的最终稳定显示仍然存在波动

### 4. 下一步方向修正

这意味着后续优化重点不应继续停留在首页首卡内部，而应转向：

1. 分析 `NewsSandBox` 首屏必经逻辑
2. 继续减少首页同步依赖的关键 chunk
3. 评估首页首屏是否仍然同步吃下过多 `antd` 组件能力
4. 继续观察 `LCP element` 与 `render delay`

## 十五、启动请求去重 + 去掉整页 Spin 后的最终结果

测试时间：2026-03-17  
优化内容：

- 将 `/api/users/me` 和 `/rights` 改为启动阶段共享缓存
- 去掉 `NewsRouter` 的整页 `Spin` 包裹，只保留右上角小 loading
- `NewsSandBox`、`SideMenu`、`NewsRouter` 复用同一份启动数据

### 1. Lighthouse 结果

本轮连续截图结果：

| 指标 | 结果 A | 结果 B |
|------|--------|--------|
| Performance | 98 | 99 |
| FCP | 0.7s | 0.7s |
| LCP | 0.9s | 0.9s |
| TBT | 0ms | 0ms |
| CLS | 0 | 0 |
| Speed Index | 1.0s | 0.9s |

### 2. 结论

这轮优化可以判断为**明确有效**：

- 首页首屏开始出现内容更稳定，`FCP` 降到 `0.7s`
- 最大内容绘制稳定在 `0.9s`
- `TBT` 继续保持 `0ms`
- `CLS` 降到 `0`

其中 `CLS = 0` 很关键，说明首屏内容在渲染期间几乎没有发生可感知布局跳动。

### 3. 原因归纳

这次结果改善，主要不是因为继续压缩某个单独 bundle，而是因为首屏链路更干净了：

1. 去掉整页 `Spin`
   - 避免请求状态变化时对整个首屏区域造成遮罩和重绘干扰

2. 启动请求去重
   - 避免 `NewsSandBox`、`SideMenu`、`NewsRouter` 各自重复请求权限数据
   - 减少首页壳层进入时的同步依赖链

3. 首卡稳定化继续发挥作用
   - 统计区布局更稳定
   - 数值回填不再显著影响首屏布局

### 4. 当前阶段判断

到这一轮为止，首页性能已经进入一个较稳定的状态：

- `Performance` 基本稳定在 `98 ~ 99`
- `FCP / LCP / CLS / TBT` 都进入较理想区间

因此，当前首页性能优化可以先告一段落。

后续如果继续优化，更适合转向：

1. 构建产物体积治理
2. 大体积 chunk 的长期拆分策略
3. 编辑器等非首页场景的按需加载
4. 面向真实业务数据增长的接口层优化
