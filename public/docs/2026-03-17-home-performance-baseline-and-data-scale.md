## 四、首页聚合接口下沉后的结果

测试时间：2026-03-17  
优化内容：

- 首页从 4 个接口请求改为 1 个聚合接口：`GET /api/dashboard/home`
- 后端统一返回：
  - 总浏览量
  - 新闻总数
  - 总点赞数
  - 用户总数
  - 热门浏览
  - 热门点赞
  - 分类分布
  - 当前登录用户个人统计

### 1. 优化后 Lighthouse 数据

| 指标 | 优化后 |
|------|--------|
| Performance | 98 |
| FCP | 0.7s |
| LCP | 1.0s |
| TBT | 0ms |
| CLS | 0.022 |
| Speed Index | 1.0s |

### 2. 结论

- 首页聚合接口下沉是一次**确定有效**的优化
- 它的核心收益不是继续压缩首包，而是减少首页首屏阶段的：
  - 请求数量
  - 返回数据体积
  - 前端全量聚合计算
- 在当前 `2 万+` 新闻数据规模下，这次改造依然让首页从 `97` 提升到 `98`
- 这说明当前项目后续的性能优化重点，应该更多放在：
  - 数据获取策略
  - 前端聚合下沉
  - 渲染时机控制
  
阶段判断：

- **应当保留** 首页聚合接口下沉这次改动
- 它比“继续微调分包”更符合当前项目的真实性能瓶颈

## 五、图表延后初始化 + 骨架屏实验记录

测试时间：2026-03-17  
实验目标：

- 让首页先渲染用户卡、统计卡、热门列表
- 折线图和分类图延后初始化
- 在图表区域加入骨架屏占位

### 1. 实验结果

| 指标 | 实验结果 |
|------|----------|
| Performance | 93 |
| FCP | 1.0s |
| LCP | 1.5s |
| TBT | 0ms |
| CLS | 0.001 |
| Speed Index | 1.3s |

### 2. 与聚合接口下沉版本对比

| 指标 | 聚合接口下沉后 | 图表延后 + 骨架屏 | 变化 |
|------|----------------|-------------------|------|
| Performance | 98 | 93 | 下降 5 分 |
| FCP | 0.7s | 1.0s | 变差 0.3s |
| LCP | 1.0s | 1.5s | 变差 0.5s |
| TBT | 0ms | 0ms | 持平 |
| CLS | 0.022 | 0.001 | 略好 |
| Speed Index | 1.0s | 1.3s | 变差 0.3s |

### 3. 结论

- 这次实验属于负优化，不应保留
- 问题不在“图表可以后置”这个方向本身，而在于当前实现方式引入了**大面积首屏骨架占位**
- 两块 `300px` 高的图表骨架进入首屏可见区域后，反而干扰了首屏主要内容的稳定展示，导致：
  - FCP 变慢
  - LCP 变慢
  - Speed Index 变差

实验结论：

- 当前项目不适合继续保留“大面积骨架 + 图表延后初始化”这版实现
- 后续如果继续优化图表，更合适的方向应是：
  1. 图表库代码层动态导入
  2. 更轻量的占位方式
  3. 避免让大面积骨架进入首屏关键区域

## 六、当前构建产物中的 ECharts 体积记录

构建时间：2026-03-17  
构建命令：`npm run build`

### 1. 关键构建产物

| Chunk | 体积 | gzip |
|-------|------|------|
| `dist/assets/Home-09ca3556.js` | 8.77 kB | 3.37 kB |
| `dist/assets/echarts-vendor-5a19c90c.js` | 504.69 kB | 170.25 kB |
| `dist/assets/index-65f13dda.js` | 247.91 kB | 75.30 kB |

### 2. 结论

- ECharts 已经被拆成独立 chunk，没有继续污染通用主包
- 但它本身仍然是一个较大的异步 chunk：
  - 原始体积约 `504.69 kB`
  - gzip 后约 `170.25 kB`
- 对“进入首页就加载图表”的场景来说，这个体积仍然足以影响：
  - 首屏请求体积
  - 首屏解析与下载时间

判断：

- **值得继续评估动态图表库加载**
- 但更推荐的实现方式不是“大骨架 + 延后渲染”
- 更适合的方向是：
  - 将图表区拆成独立异步组件
  - 图表库在组件级动态导入
  - 保持首屏关键文本内容优先展示

## 八、后续优化方向

下一阶段如果继续优化首页，不建议只盯着分包，而应优先考虑：

1. 新增 dashboard 聚合接口，减少首页全量拉取新闻再前端聚合
2. 评估 ECharts 组件级动态导入，而不是继续使用大面积骨架屏
3. 缓存 `rights / roles / categories / users/me` 这类低频数据
4. 降低首页进入时的重复请求和重复渲染

## 九、结论

当前首页性能状态可以概括为：

- **首屏资源问题**：已经通过懒加载和分包基本解决
- **未来重点问题**：应转向“数据获取策略”和“渲染时机优化”

## 十一、首页顶部首卡为什么会成为 LCP 元素

分析对象：`src/sandbox/home/Home.tsx`

### 1. 当前首卡结构

首页顶部第一张卡片包含：

- 用户头像与用户名
- 区域与角色
- “我的新闻 / 我的浏览 / 我的点赞”三项统计
- “查看个人数据分析”入口

这块区域具备三个典型特征：

1. 它处在首屏最上方
2. 它的可见面积大于普通文本行
3. 它的统计数字依赖异步接口返回后再更新

因此 Lighthouse 很容易把它识别成首屏最大的内容元素。

### 2. 它为什么会拖慢 LCP

虽然首页已经把图表做成了异步组件，但顶部首卡仍然会受到这类问题影响：

- 进入页面先渲染一版默认值：`0`
- `/api/dashboard/home` 返回后，再把三项个人统计更新成真实值
- 对 Lighthouse 来说，LCP 关注的是“最终稳定显示”的时间，而不是“占位先出现”的时间

所以即使页面已经有基础骨架，若首卡中的主要文本块或统计块在接口返回后发生更新，`LCP element render delay` 仍然会被拉高。

### 3. 这说明当前瓶颈已经变化

现在首页的主要问题不再是：

- 打包没拆开
- 图表库阻塞主线程

而更接近：

- 首屏关键文本区依赖异步数据后置更新
- 首屏最大元素恰好位于顶部首卡
- Lighthouse 把“首卡最终稳定时间”算进了 LCP

## 十二、针对顶部首卡的具体优化点

这部分优先级应高于继续微调图表。

### 1. 优先做首卡数据稳定化

目标不是“把首卡隐藏掉”，而是让它更早稳定。

可行方向：

- 将顶部首卡拆成独立组件，减少首页其它状态更新对它的连带重渲染
- 对三项个人统计使用固定宽度占位，避免数字回填时影响文本块稳定性
- 若接口未返回，可先展示轻量占位文本，例如 `--`

注意：

- 这里更适合“小面积稳定占位”
- 不适合再引入大面积骨架屏，因为前面的实验已经证明会拉低分数

### 2. 优先缓存 dashboard 聚合结果

如果用户是从登录页进入首页，可以考虑：

- 登录成功后预取 `GET /api/dashboard/home`
- 或在首页对该接口做短时内存缓存 / session 缓存

这样可以减少二次进入首页时首卡再次从 `0` 更新到真实值的过程。

### 3. 把“用户静态信息”和“统计数据”拆开看

当前首卡里：

- 用户名、区域、角色基本是同步可得的
- 个人统计完全依赖接口

更合理的思路是：

- 用户身份区立即稳定渲染
- 统计区单独更新，但保持固定布局和固定宽度

这样即使统计值稍后回填，也不容易让整个首卡继续充当 LCP 的最终完成点。

### 4. 继续观察 LCP element，而不是只看总分

后续每次优化首页时，除了看 `Performance` 总分，还应重点确认：

- `LCP element` 是否仍然是顶部首卡
- `element render delay` 是否下降
- 是否出现新的首屏大块元素替代它成为 LCP

## 十三、下一步建议

如果继续优化首页，建议顺序如下：

1. 先处理顶部首卡的稳定渲染问题
2. 再评估 dashboard 数据缓存
3. 最后再决定是否继续细调图表加载策略

原因：

- 当前 `TBT` 已经是 `0ms`
- 继续压 JS 对首页分数的边际收益在下降
- 顶部首卡才是当前更真实的 LCP 优化目标

因此，后续首页性能优化的重点，应从“如何继续压缩首包”，转向“如何减少首页不必要的数据请求、前端聚合和首屏关键区域的延后渲染”。
